# Centinela Impeccable Standard

## Purpose

Centinela is a Paraguay public-integrity intelligence system, not a generic dashboard and not an accusation engine.

The product surface should make a serious reviewer understand the system quickly:

- what public records are being used
- which entities and relationships are visible
- what is a signal, candidate, accepted identity context, source record, evidence link, or limitation
- how to move from a dossier to a case packet to artifact/source verification
- why nothing on the screen is proof of guilt

## Users

Primary users are analysts, researchers, civic-tech builders, and reviewers who need source-backed public-integrity leads without losing provenance or caution.

They may not know the internal commands. The interface should therefore give them one calm path through the evidence spine before exposing advanced controls.

## Product Personality

Serious, grounded, careful, and precise.

Centinela should feel like a disciplined public-records workbench with an investigative memory, not like a flashy risk-score product. It should be confident about process and humble about conclusions.

## Design Principles

1. Dossier before dashboard.
   The first useful action should be opening a real entity dossier, not staring at abstract metrics.
2. Evidence trail over spectacle.
   Show the path from entity to source records, case packet, artifact bundle, and verification checks.
3. Review-first language everywhere.
   Use "lead", "signal", "candidate", "identity context", "source limitation", and "requires review"; never imply guilt or legal conclusion.
4. Progressive disclosure.
   Keep the main path readable. Put raw JSON, advanced write controls, and dense technical payloads behind clear workbench sections.
5. No fake proof.
   Do not invent example data, source records, matches, or public claims. Use live data or say the surface is waiting for data.
6. Source-grounded humility.
   Make provenance, source URLs, retrieval state, hash checks, access failures, and public-safety gates visible without making the interface defensive.
7. Product-like, not decorative.
   Visual polish must clarify workflow and trust. Avoid filler cards, noisy gradients, generic AI copy, and ornamental metrics.

## Interface Gates

Before treating a Command Center change as done, verify:

- the first viewport gives a clear "what this is" and "what to do first"
- the guided path can open a live entity, case packet, artifacts, and verification context
- accepted matches remain separated from review candidates and risk signals
- public-safety and limitation language is visible near evidence surfaces
- raw JSON is available but not the only way to understand a case
- the local smoke test still passes

## Current Canonical Proof Path

Use `CONSULTORA GUARANI SA INGENIEROS CIVILES` as the default guided path because it demonstrates:

- DNCP/DNIT local identity context
- accepted external identity context through second review
- official IDB row-level source evidence
- source-pack case `20`
- source-document index matches
- artifact/source verification checks

This path is still internal review context, not a public finding.
