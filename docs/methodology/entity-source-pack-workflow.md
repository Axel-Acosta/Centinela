# Entity Source Pack Workflow

## Purpose

The entity source pack workflow turns entity-linked source records into a usable analyst case packet.

It is designed for companies or people that already have source records from DNCP release checks, document-content capture, external source checks, or later cross-domain connectors. It does not decide wrongdoing. It organizes evidence, provenance, limitations, and local source files so an analyst can review the entity without hunting through raw database rows.

## Command

```bash
npm run database:entity-source-pack -- --entity-name "Entity Name" --source-record-limit 10 --source-index-query "search terms"
```

Use the readiness report before widening source packs:

```bash
npm run database:entity-source-pack-readiness -- --limit 25 --source-record-limit 10
```

The readiness report ranks high-priority company entities and shows whether the next action is a DNCP release source check, source-pack build, selective document-content capture, or internal review with documented download/parser limits.

The same workflow is also available through the local internal console/API:

```text
POST /api/entities/:id/source-packs
```

The API route requires `CENTINELA_WRITE_TOKEN`. The console exposes both preview and write buttons in the case workbench after an entity is open.

Useful options:

- `--entity-id <id>` selects an entity by database ID instead of name.
- `--case-id <id>` adds source-pack links into an existing analyst case.
- `--case-key <key>` reuses a stable case key so reruns do not create a new case.
- `--record-kinds document_content_extract,ocds_document_metadata` limits selected source-record kinds.
- `--query contrato` filters selected source records by payload text.
- `--public-only true` uses the existing public-safety gate and only works after `approved_public`.
- `--copy-assets false` writes bundle metadata without copying local source files.
- `--dry-run true` writes a source-pack report without changing case/evidence tables.

## What It Creates

When not in dry-run mode, the workflow:

- creates or reuses an analyst case centered on the selected entity
- links the entity as the main case target
- links selected source records as case targets
- creates source-record evidence links against the entity with field path, field value, source summary, limitations, evidence role, and metadata
- skips evidence links already generated by the entity source pack for the same case/entity/source record
- writes a case evidence artifact
- writes a source attachment manifest
- writes a source bundle with copied source-run assets where paths resolve
- refreshes a source-document index for the bundle
- writes a Markdown and JSON source-pack summary under the local runtime output folder

## Evidence Semantics

Entity source packs are source-backed review packets. They are not findings, legal conclusions, sanctions determinations, or proof of wrongdoing.

Evidence links are intentionally conservative:

- `document_content_extract` records cite official document capture status and SHA-256 hash.
- `document_content_extract` records can also cite failed download attempts, such as official DNCP document URLs returning `404`; these are source-access limitations, not evidence about the entity.
- `ocds_document_metadata` records cite official document metadata and process context.
- `ocds_release_package` records cite official DNCP OCDS release package context.
- Parser limitations such as `no_extractable_text` are preserved as source/document limitations, not treated as missing evidence about an entity.

## Reference Influence

- br/acc shapes the source-registry discipline: every packet starts from source records and preserves source keys, external IDs, URLs, and retrieval context.
- Aleph shapes the casework pattern: source records become case evidence and searchable document bundles.
- Sayari shapes the entity-intelligence direction: a company/person can accumulate source-backed identity and relationship context across sources.
- QuiénEsQuién/TodosLosContratos shapes the contracts-plus-company accountability path: procurement/company source context becomes explorable from the entity.
- DNCP/OCDS shapes the current first source family: release packages, document metadata, and captured official documents stay tied to official procurement records.
- RUBLI and Integrity Watch shape the language: limitations, public-safety gates, and non-accusatory wording remain explicit.

## Current Limits

- The current source-record selector is strongest for records that include `payload.centinelaTarget.entity_id`.
- Existing captured DNCP PDFs may be scanned/image-only; OCR is not automatic yet.
- Public-only exports still require explicit `approved_public` case review.
- Source bundles live in the local runtime output folder, not Git, and should not be treated as publication-ready.

## Next Improvements

- Add OCR only for case-priority scanned PDFs or after a successful official document download justifies it.
- Investigate alternate lawful DNCP document access only if repeated official document URLs return `404` for a case-priority source pack.
- Extend source selection to external candidate and accepted-match source records when they are linked to the same entity.
- Add a durable artifact registry entry for entity source packs if analysts start using them as a recurring workflow.
